## CẤU HÌNH KIỂM SOÁT CẤU PHẦN (TUỲ BIẾN - ĐỌC TRƯỚC KHI THỰC THI)

> **Hướng dẫn sử dụng:** Chỉ khai báo các tham số bạn muốn thay đổi. Bỏ trống = giữ nguyên giá trị mặc định (DEFAULT). Đặt block này ngay tại đây trước khi chạy lệnh.

```
MANDATORY_CHECK : [DEFAULT]
# Giá trị hợp lệ : DEFAULT | BOTH | MISSING_ONLY | EMPTY_ONLY | OFF
# DEFAULT        : Sinh đủ cả 2 case Missing key và Empty value cho mọi field Required=Y
# BOTH           : Tương đương DEFAULT
# MISSING_ONLY   : Chỉ sinh case thiếu key, bỏ qua Empty
# EMPTY_ONLY     : Chỉ sinh case truyền rỗng, bỏ qua Missing
# OFF            : Bỏ qua toàn bộ check bắt buộc

TYPE_CHECK      : [DEFAULT]
# Giá trị hợp lệ : DEFAULT | ON | STRICT | OFF
# DEFAULT        : Sinh 1 case sai kiểu cơ bản cho mỗi field (VD: String thay vì Integer)
# ON             : Tương đương DEFAULT
# STRICT         : Sinh thêm case sai kiểu thứ 2 (VD: Boolean, Array, Object tuỳ kiểu gốc)
# OFF            : Bỏ qua toàn bộ check kiểu dữ liệu

LENGTH_CHECK    : [DEFAULT]
# Giá trị hợp lệ : DEFAULT | MAX_ONLY | MIN_ONLY | BOTH | OFF
# DEFAULT        : Chỉ sinh case vi phạm Max Length (N+1 ký tự) khi tài liệu có con số cụ thể
# MAX_ONLY       : Tương đương DEFAULT
# MIN_ONLY       : Chỉ sinh case vi phạm Min Length (N-1 ký tự) khi tài liệu có con số cụ thể
# BOTH           : Sinh cả Max (N+1) và Min (N-1) nếu tài liệu ghi đủ cả 2 giá trị
# OFF            : Bỏ qua toàn bộ check độ dài

SCOPE_FIELDS    : [DEFAULT]
# Giá trị hợp lệ : DEFAULT | [field_1, field_2, ...]
# DEFAULT        : Duyệt tuần tự toàn bộ field trong Request Payload (trừ Header)
# [field_1, ...] : Chỉ chạy vòng lặp trên các field được liệt kê, bỏ qua phần còn lại
# Ví dụ          : SCOPE_FIELDS: [amount, currency, receiver_account]
```

> **Quy tắc áp dụng bắt buộc:**
> - Đọc và áp dụng toàn bộ tham số trên **TRƯỚC** khi bắt đầu Thuật toán tư duy (Mục III).
> - Tham số `[DEFAULT]` = hành vi gốc của prompt, không thay đổi bất kỳ logic nào phía dưới.
> - Tham số khác `[DEFAULT]` = ghi đè đúng bước tương ứng trong Thuật toán tư duy.
> - Nếu `SCOPE_FIELDS` khác DEFAULT: bỏ qua mọi field không có trong danh sách, kể cả khi field đó là Required.

## I. MỤC TIÊU VÀ ĐỘ PHỦ (CHỈ TẬP TRUNG SCHEMA)

Chỉ kiểm thử Cấu trúc dữ liệu (Schema) của Request Body/Params dựa trên bảng PTTK:
- [Missing] / [Empty] Mandatory Check: Check sự tồn tại theo cột "Required/Bắt buộc". CHỈ TẠO 2 case: Missing key và Empty value. (TUYỆT ĐỐI KHÔNG TẠO CASE TRUYỀN NULL).
- [Type] Data Type Check: Check sai kiểu dữ liệu (VD: String thay vì Integer, Object thay vì Array) theo cột "Kiểu dữ liệu".
- [Max Length] Length Constraint: Check vi phạm độ dài chuỗi ký tự theo cột "Max Length" (VD: Nhập N+1 ký tự). CHỈ CHECK nếu CÓ con số cụ thể.

## II. LỆNH CẤM (CONSTRAINTS - NGHIÊM NGẶT)

1. KHÔNG sinh test case liên quan đến HEADER (Authorization, Token...).
2. KHÔNG TẠO case truyền null khi check Mandatory. Chỉ dùng Missing và Empty.
3. KHÔNG TỰ BỊA / ASSUME con số Max Length nếu tài liệu không ghi. (Không ghi = Bỏ qua check length).
4. KHÔNG test logic nghiệp vụ (Số âm, min/max value, business rules) hay logic chéo ở prompt này.
5. KHÔNG Verify DB cho các case Validation lỗi (HTTP 400). Chỉ Verify DB cho case Happy Path (TD_001).

## III. THUẬT TOÁN TƯ DUY (INTERNAL ALGORITHM)

Chạy ngầm quy trình sau trước khi xuất kết quả:
- Bước 0: Đọc toàn bộ CẤU HÌNH KIỂM SOÁT CẤU PHẦN ở trên. Ghi nhớ giá trị từng tham số.
- Bước 1: Sinh TD_001 Happy Path (Valid Schema) - Request chứa đầy đủ các trường đúng định dạng.
- Bước 2: Xác định danh sách field cần duyệt:
  + Nếu `SCOPE_FIELDS = DEFAULT`: lấy toàn bộ field trong Request Payload (bỏ Header).
  + Nếu `SCOPE_FIELDS` có danh sách cụ thể: chỉ duyệt các field trong danh sách đó.
- Bước 3: Vòng lặp Field-by-Field. Duyệt tuần tự từng field (F) trong danh sách đã xác định ở Bước 2. Với mỗi field, thực hiện ĐÚNG THỨ TỰ và ÁP DỤNG tham số kiểm soát:

  + **[Mandatory Check]** — Áp dụng `MANDATORY_CHECK`:
    - `DEFAULT` hoặc `BOTH`: Sinh Missing key VÀ Empty value (nếu field Required=Y).
    - `MISSING_ONLY`: Chỉ sinh Missing key (nếu field Required=Y).
    - `EMPTY_ONLY`: Chỉ sinh Empty value (nếu field Required=Y).
    - `OFF`: Bỏ qua, không sinh case nào.

  + **[Type Check]** — Áp dụng `TYPE_CHECK`:
    - `DEFAULT` hoặc `ON`: Sinh 1 case sai kiểu cơ bản.
    - `STRICT`: Sinh 2 case sai kiểu (cơ bản + thêm 1 kiểu sai khác phù hợp với kiểu dữ liệu gốc).
    - `OFF`: Bỏ qua, không sinh case nào.

  + **[Length Check]** — Áp dụng `LENGTH_CHECK` (chỉ khi tài liệu có con số cụ thể):
    - `DEFAULT` hoặc `MAX_ONLY`: Sinh case N+1 ký tự nếu tài liệu ghi Max Length.
    - `MIN_ONLY`: Sinh case N-1 ký tự nếu tài liệu ghi Min Length.
    - `BOTH`: Sinh cả N+1 (nếu có Max) và N-1 (nếu có Min).
    - `OFF`: Bỏ qua, không sinh case nào.

  + BẮT BUỘC xong field F mới chuyển sang field tiếp theo.

## IV. VÍ DỤ MẪU OUTPUT (GOLDEN SAMPLE)

Bắt buộc xuất theo format Markmap như ví dụ (chú ý giữ đúng các ký tự #, ##, ###, -):

```
## Schema Validation
### TD_P2_001 - [ST] - Happy Path Full Flow (Valid Schema)
- **Steps**: Request body với đầy đủ thông tin hợp lệ.
- **Expected**: HTTP 200, Code 'SUCCESS'. DB lưu 1 record mới.
<!-- BẮT ĐẦU VÒNG LẶP FIELD: 'amount' (Type: Integer, Required: Y, Max-len: Không có) -->
### TD_P2_002 - [Missing] - Field 'amount' bị Missing
- **Steps**: Request body thiếu key 'amount' (Check Required).
- **Expected**: HTTP 400, Code 'ERR_MISSING_FIELD'.
### TD_P2_003 - [Empty] - Field 'amount' truyền rỗng
- **Steps**: Request với 'amount' = "" (Check Empty).
- **Expected**: HTTP 400, Code 'ERR_MISSING_FIELD'.
### TD_P2_004 - [Type] - Field 'amount' sai kiểu dữ liệu (String)
- **Steps**: Request với 'amount' = "một triệu" (Check Type).
- **Expected**: HTTP 400, Code 'ERR_INVALID_TYPE'.
<!-- Lặp tiếp cho các field khác, ID NNN tiếp tục tăng dần... -->
```

## V. THỰC THI CUỐI

1. Tự Self-Audit: Đã xóa hết case test Header chưa? Có case Null nào không? Có tự bịa Max Length không? Có sinh case của field ngoài SCOPE_FIELDS không?
2. Rendering: Xuất kết quả dưới dạng MỘT FILE MARKDOWN DUY NHẤT nằm trong code fence. KHÔNG in thêm bất kỳ văn bản/bảng biểu nào khác ngoài luồng.
