# Legacy API/UI Manual Test Case 19-Column Contract

Header legacy ưu tiên cho API/UI manual testcase export legacy:

```tsv
"Test Case ID"\t"Function"\t"Group Tests"\t"Scenario Outline"\t"Test Case Summary"\t"Pre-conditions"\t"Test Datas"\t"Test Steps"\t"Expected result"\t"Environment"\t"Priority"\t"Regression"\t"Automation"\t"Manual Test Results Round 1"\t"Manual Test Results Round 2"\t"Automation Test Results"\t"Actual result"\t"BugID"\t"Notes"
```

## Scope

- Applies to API/UI manual testcase exports generated by workflow-pack routes.
- Sample testcase sheets may contain overflow columns beyond the 19-column contract, including observed 49/67/139-column exports. Report overflow columns; do not treat them as contract expansion.
- TSV export must preserve exactly 19 columns.

## Header aliases

| Alias | Canonical |
|---|---|
| `Expected / result` | `Expected result` |
| `Precondition` | `Pre-conditions` |
| `Test Data` | `Test Datas` |
| `Step` | `Test Steps` |

## Column quality rules

| Column | Requirement |
|---|---|
| Test Case ID | Stable unique ID; should reflect happy/unhappy/boundary/role/state when naming convention supports it. |
| Function | Business function, API operation, or UI function under test. |
| Group Tests | Happy Path, Unhappy Path, Boundary, Permission, State, Regression, or other meaningful group. |
| Scenario Outline | Main scenario from Test Condition Summary / TD node. |
| Test Case Summary | One primary condition + one expected outcome; avoid combining unrelated branches. |
| Pre-conditions | Env, role/user, permission/scope, URL/screen/endpoint, token/session/header, existing data state, config/dependency, pre-data. Missing facts must use `[PENDING_DOC:<fact>]`. |
| Test Datas | Concrete data values, not generic words like “valid data”; include field=value, boundary class, enum/status, or payload snippet when applicable. |
| Test Steps | Executable steps with action/input/verify. API steps must include method/path/header/payload mutation/execute/verify response; UI steps must include screen/control/action/input/verify. |
| Expected result | Map 1-1 with verify steps; include status code/message/response fields/UI state/data state. DB verify only when source/rule allows it. |
| Environment | SIT/UAT/ALL or `[PENDING_DOC:environment]`. |
| Priority | High/Medium/Low. |
| Regression | Yes/No or blank if not applicable. |
| Automation | Candidate/Manual/No or blank if not assessed. |
| Manual Test Results Round 1 | Blank before execution. |
| Manual Test Results Round 2 | Blank before execution. |
| Automation Test Results | Blank before execution. |
| Actual result | Blank before execution. |
| BugID | Blank before execution. |
| Notes | Empty unless there is an explicit open question/rationale required by the route. |

## Format rules

- Mỗi testcase là một dòng duy nhất.
- Delimiter là tab, mỗi dòng đúng 18 tab.
- Quote all cells.
- Escape `"` thành `""`.
- Newline logic trong cell dùng `\n`, không xuống dòng vật lý.
- Test Case ID API giữ prefix markdown: `TD_P1_001_TC_001`.
- Với API, counter reset theo group cấu phần `TD_P1`, `TD_P2`, `TD_P3`.

## Dense coverage requirements

- Testcase generation must be driven by `CoverageMatrix.md` / `TestGenerationMatrix.md` when the route produces TD/TC artifacts.
- Every source rule, TD node, API field, UI validation, business rule, permission rule, and state rule must be considered for expansion.
- Apply ECP, BVA, Decision Table, State Transition, Error Guessing, Pairwise/Cross-field, Permission/Role, and Regression/Smoke classification when evidence supports them.
- Split testcase rows by primary condition and expected outcome; do not hide multiple branches in one row.
- Missing expected behavior must be represented as open question or `[PENDING_DOC:<fact>]`, not guessed.

## Step/precondition specificity gate

A testcase is too vague and must be rewritten when:

- Pre-conditions do not identify role/environment/data state/config needed to execute.
- Test Data says only “valid”, “invalid”, “correct”, “data chuẩn”, or “như trên” without concrete values/classes.
- Test Steps do not name the exact field/header/control/rule being tested.
- Expected result does not include a measurable status/message/state/field value.
- One testcase contains multiple independent negative paths or expected outcomes.
